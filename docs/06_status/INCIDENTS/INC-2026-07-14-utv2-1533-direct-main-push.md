## Severity justification

**Raised to `High` governance-control risk / `Low` production impact** (PM decision, 2026-07-15), from the original `Medium`:

- **Production impact remains `Low`**: across all three occurrences, every pushed diff was truthful, additive, control-plane/proof bookkeeping only — no code, schema, migration, or runtime behavior change; no data loss; no outage; no false evidence. This dimension has not changed.
- **Governance-control risk is now `High`**, not `Medium`: the identical prohibited action — a direct, un-reviewed push to `main` editing protected operational-truth files to make a lane appear closed, with no emergency-exception record — has now happened **three times** (2026-07-10, 2026-07-14, 2026-07-15), not two. ## Root Cause

**Immediate cause:** `pnpm ops:lane-close UTV2-1533 --repair-merged`, run automatically by `post-merge-lane-close.yml` on the `push` event, correctly fail-closed post-merge because PR #1216 merged with `docs/06_status/proof/UTV2-1533/evidence.json` missing the `verifier`/`runtime_proof` sections that `scripts/ops/truth-check-lib.ts`'s `runtime_proof_required` gate requires unconditionally for `tier === 'T1'` (checks C6, P7, P9, P10, R1, R2, R3). This part of the system behaved correctly — it is a fail-closed gate, and it did not silently pass.

**Contributing cause 1 — no governed low-friction repair path existed.** The only existing automated mechanism that writes directly to `main` (`ops:proof-generate --merge-sha`, invoked by this same workflow using the `SYNC_BOT_TOKEN` PAT, an already-documented and narrowly-scoped bot exception — see the workflow's own header comments) can only rebind an already-known, externally-immutable merge SHA into existing proof files (`rebindEvidenceJsonSha`/`rebindVerificationMdSha` in `scripts/ops/proof-generate.ts`, both additive/patch-only on the `sha_binding` key). It has no capability to add `runtime_proof`/`verifier` evidence — a categorically different, evidentiary (not merely referential) class of claim. When that gate failed, the workflow's own remediation text ("...re-run the workflow manually or push a new commit") did not name a governed path for adding the missing evidence, and no script existed to scaffold one. Faced with a failing gate and no scripted alternative, a human operator (not the bot identity) edited the proof files directly and pushed to `main`.

**Contributing cause 2 — the push used an explicit ruleset bypass, not just an unenforced admin exemption.** Two distinct pieces of evidence, corroborating and refining each other:

1. This lane's own tracking issue (UTV2-1537, authored by the repository owner, `griff843`) quotes the literal git server response captured at push time:
   ```
   remote: Bypassed rule violations for refs/heads/main:
   remote: - Changes must be made through a pull request.
   remote: - 4 of 4 required status checks are expected.
   ```
   This is GitHub's exact wording for a **repository ruleset** bypass (the newer rules engine, distinct from classic branch protection) — it confirms the push was not merely "unenforced," it explicitly invoked a bypass the pushing identity was permitted to use, and GitHub logged that fact back to the client at push time.
2. Verified independently via `gh api repos/griff843/Unit-Talk-v2/branches/main/protection` (the classic branch-protection API, which can coexist with rulesets): `required_status_checks` names `verify`, `Executor Result Validation`, `Merge Gate`, `P0 Protocol` as required contexts, `required_pull_request_reviews.required_approving_review_count` is `0`, and `enforce_admins.enabled` is `false` — meaning even independent of any ruleset, classic protection also does not block a repository-admin identity's direct push.

`griff843` (the commit's author) is the repository owner, so both mechanisms converge on the same outcome: this identity's direct push to `main` was permitted, not merely unenforced-by-oversight.

**What I could not independently confirm right now:** re-querying `gh api repos/griff843/Unit-Talk-v2/rulesets` and `gh api repos/griff843/Unit-Talk-v2/rules/branches/main` at the time this record was authored both returned an empty array — no ruleset currently appears to apply to `main`. I cannot tell from here whether the ruleset that produced the quoted bypass message has since been removed/reconfigured, or whether my token/scope simply cannot enumerate it. I am stating the quoted push-time evidence as reported in the issue record (highly specific, technically exact GitHub server wording, authored by the repository owner) without independently re-deriving the ruleset's current configuration, and I am not claiming to have verified the ruleset's present state. GitHub's private audit log — the only place this could be fully settled — is not accessible to this investigation.

**Contributing cause 3 — occurrence 2 was an unlogged recurrence of occurrence 1.** The identical pattern (a direct commit to `main` editing lane/proof bookkeeping to make a lane appear closed, with no emergency-exception record) occurred previously on `2026-07-10T15:35:41Z` as commit `4fe0107551dfc67f09e4c954fa7edb6693d4367f` ("chore(lanes): park UTV2-1512/1513 during OS v1 stabilization wave", no associated PR), part of what later became UTV2-1519 ("revert direct-main park bypass") — see Occurrence 1 under Timeline for the full evidence, confirmed via `gh pr view 1185` (body explicitly states: *"That direct push matched the exact prohibited pattern in docs/05_operations/DIRECT_MAIN_BYPASS_POLICY.md ... without recording an emergency-exception record. This was ordinary bookkeeping, not an emergency; no retroactive exception is being recorded (PM decision)."*). That occurrence was remediated (revert PR #1185 + re-land PR #1186) but **was never logged in `docs/06_status/INCIDENTS/`** — the only pre-existing entry in the index at the time was the unrelated `INC-2026-04-10-utv2-519`. Without a durable, discoverable incident record from occurrence 1, there was no artifact for the operator (or any future operator) to find and be reminded of before repeating the same pattern four days later as occurrence 2.

**Contributing cause 4 — `ops:lane-close --repair-merged` had no governed landing path, unlike `proof-repair.ts`, and this is what actually produced occurrence 3.** `proof-repair.ts` (built in this same lane, for occurrence 2) has an explicit design contract: never runs `git push`, never targets `main`, and its `scaffold` command prints the exact branch/PR steps. `--repair-merged` — the *other* pre-existing repair tool, used for manifest/truth-check reconciliation rather than missing runtime evidence — had no equivalent contract. Run from the shared root checkout (as its own design implicitly invites: it operates on `docs/06_status/lanes/<ISSUE>.json` in the current working tree, with no worktree/branch requirement), it leaves a repaired, tracked-file-modified manifest sitting in whatever checkout it was run from, with **zero output signal** that the result must be landed via PR rather than committed in place. Faced with a legitimate, verified-correct repair and no tool-provided next step, the operator did the naive thing: committed and pushed it directly, exactly reproducing the same violation `proof-repair.ts` was built to prevent for a different trigger. This gap is closed in this same PR (see Remediation): `guardRepairAgainstMainCheckout()` now intercepts this exact condition and blocks the write, emitting a `repair_required_via_pr` result with the governed branch/PR steps instead.

## Policy / Control Failure

`docs/05_operations/DIRECT_MAIN_BYPASS_POLICY.md` (ratified, Active, UTV2-1432) already prohibits exactly this: *"Editing protected operational truth files on main to make a lane appear closed"* is explicitly listed under Prohibited Bypasses. The policy itself is sound and was not ambiguous. What failed was enforcement and tooling around it:

1. **No mechanical detector** existed to notice, after the fact, that a commit on `main` had no associated PR and no documented emergency exception. The policy lived only in prose with no CI or script checking compliance against real git/GitHub history — a violation of this repo's own invariant #11 ("If a rule can be enforced mechanically, it must not live only in prose"). This lane's Half 2 work (`scripts/ci/direct-main-push-guard.ts`) closes that specific gap.
2. **No incident log entry** was created for the July 10 precedent (UTV2-1519), so the corrective memory that should have existed after the first occurrence did not exist in a place a future operator would consult before making the same choice again.
3. **No governed repair-path script** existed for the single most common trigger of this exact bypass shape: a T1 lane merging without runtime proof and the workflow's own remediation message not steering the operator toward "open a small PR" as the objectively-easier next step. This lane's Half 2 work (`scripts/ops/proof-repair.ts` + an updated remediation message in `post-merge-lane-close.yml`) closes that gap.
4. **The hosting-side bypass configuration (a ruleset bypass, plus `enforce_admins: false` on classic branch protection) was not itself examined as a contributing factor** until this incident's investigation. This record surfaces it; changing GitHub-side settings is an account/repo-configuration action for the repository owner, outside the scope of a code lane, and is called out under Prevention / Lessons below rather than changed unilaterally by this PR. **Update (occurrence 3, PM decision):** `enforce_admins` will NOT be enabled today — see Prevention / Lessons for the "prepare, then enable" sequencing and the concrete blockers (`SYNC_BOT_TOKEN` closeout automation, the `--admin` merge-gate escape hatch) that must be resolved first.
5. **`ops:lane-close --repair-merged` had no equivalent governed-landing contract to `proof-repair.ts`'s** — this is what actually produced occurrence 3, and was not examined as a distinct gap from contributing-cause-1 until occurrence 3 happened. Closed in this same PR — see Contributing cause 4 and Remediation.

## Remediation

Implemented in this lane (UTV2-1537) — see this record's Fix PR field once opened:

- `scripts/ops/proof-repair.ts` — a new, additive-only, governed repair mechanism for adding missing T1 runtime evidence (`verifier`/`runtime_proof`) to an already-merged lane's proof bundle, designed to be run on a fresh branch and landed through a normal PR — never to write to `main` directly. It refuses to run without a real, previously-captured `pnpm test:db` result file; it refuses to overwrite an `sha_binding.merge_sha` that is already correctly set; it only merges in the specific missing keys, leaving all hand-authored narrative sections byte-identical.

## Prevention / Lessons / New Controls

- **Mechanical detector added** (`scripts/ci/direct-main-push-guard.ts`, this lane): classifies commits on `main` from repo/GitHub-API-visible signals only. Its honest capability boundary — stated plainly rather than oversold — is documented in its own module header and in this lane's verification notes: it can reliably tell (a) whether a commit has an associated merged PR via `gh api commits/{sha}/pulls`, (b) whether the committer identity matches a known, allow-listed automation identity/commit-message pattern, and (c) whether a same-day incident doc references the SHA with the policy's required emergency-exception fields. It **cannot** verify GitHub's private audit log, cannot distinguish "an admin push succeeded because no rule applied" from "an admin explicitly invoked a bypass affordance," and cannot itself change branch-protection configuration. **Confirmed against occurrence 3** (see Detection Path): its classification logic is correct for this exact push; the only gap was that this workflow was not yet merged to `main` when occurrence 3 happened.
- **Governed repair path added for missing runtime evidence** (`scripts/ops/proof-repair.ts`, this lane): closes the specific "no low-friction alternative to direct-main-edit" gap that this incident's contributing-cause-1 identifies. Never writes to `main`; produces a diff meant for a small, normal PR.
- **Governed repair path added for manifest/truth-check reconciliation** (`scripts/ops/lane-close.ts`'s `guardRepairAgainstMainCheckout()`, this lane, added for occurrence 3): closes the distinct gap contributing-cause-4 identifies — the exact tool and exact shared-checkout condition that produced occurrence 3.
- **`enforce_admins` decision (PM, post-occurrence-3): prepare, then enable — not enabled today.** Turning it on immediately would trade "a human can bypass in a lapse" for "closeout automation silently breaks": `SYNC_BOT_TOKEN` (used by `post-merge-lane-close.yml`'s bot closeout commits and `proof-generate.ts`'s SHA-rebind commits) is explicitly documented in-repo as "a PAT from the repo owner (admin bypass on classic branch protection)" — the exact same admin exemption occurrence 3 used — and `gh pr merge --admin` is a real, currently-used escape hatch (documented in `docs/06_status/OS_V1_LOCK.md` and multiple T2 verification logs) for a still-not-fully-root-caused stale/duplicate check-run bug in `merge-gate.yml`. The target end state, gated behind a new dedicated T1 prerequisite program: replace owner-PAT admin bypass with either a GitHub App scoped via a ruleset bypass actor for narrowly-defined closeout paths, or governed automatic repair/closeout PRs that need no direct-main writer at all; root-cause and eliminate the stale/duplicate check-run condition; prove the full matrix (ordinary merges, proof binding, lane-close automation, bot repair, human direct-push rejection, emergency rollback) via staging/dry-run evidence; only then set `enforce_admins: true`, with a verified, data-safe rollback path (a single branch-protection API toggle, no history rewrite either direction).
- **Recommend:** every future direct-main-bypass discovery gets a `docs/06_status/INCIDENTS/` entry immediately, even when (like UTV2-1519 originally was) it is remediated same-day and feels "closed" — the absence of that entry for UTV2-1519 was a confirmed contributing factor in occurrence 2, and occurrence 3 shows a *merged* record isn't even required for the value of "immediately" — an **open, in-review** record for occurrence 2 did not prevent occurrence 3, which argues for merging detection/prevention lanes with real urgency once opened, not treating them as routine backlog once the write-up exists.
- **Temporary containment (effective immediately per PM, until `enforce_admins` lands):** no agent may execute `git push origin main`; all pushes target a non-`main` lane branch; `ops:merge-wrapper` is the only ordinary merge path; any `ops:lane-close --repair-merged` result that changes tracked files must stop and open a repair PR (mechanically enforced by this PR's `guardRepairAgainstMainCheckout()` for the manifest-reconciliation case); no unattended `--dangerously-skip-permissions` session may remain active after its assigned lane ends; verify the current branch is not `main` before any mutation.
