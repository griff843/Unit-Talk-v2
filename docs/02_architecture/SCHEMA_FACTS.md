# Schema Facts

**Status:** Canonical
**Purpose:** Non-obvious, trip-worthy schema naming facts that are stable, frequently needed, and must not live in root instructions or agent memory.
**Out of scope:** volatile snapshot facts (migration head counts, row counts, recently-added columns). For those, query Supabase directly via `list_migrations`, `list_tables`, or `describe table`.

This document exists because several column names and semantics in the Unit Talk V2 schema are intuitive-sounding but wrong. Guessing them causes broken queries, silent mismatches, and wasted diagnostic cycles. If you are writing SQL or TypeScript that references any of the facts below, read them first; do not guess.

Authoritative runtime schema truth: `packages/db/src/database.types.ts` (generated by `pnpm supabase:types`).

---

## 1. `picks.status` — not `lifecycle_state`

The lifecycle state column on `picks` is **`status`**, not `lifecycle_state`.

- Column: `picks.status`
- Type: text enum-like (values from the canonical lifecycle union)
- Allowed values: `validated | queued | posted | settled | voided`
- Contract source: `packages/contracts/src/submission.ts` (lifecycle union) and `packages/db/src/lifecycle.ts` (transition enforcement)

There is **no** `picks.lifecycle_state` column. Any reference to `lifecycle_state` in queries or code is a bug.

Related:
- `pick_lifecycle` is the canonical per-transition event table. It is **not** named `pick_lifecycle_events` and there is no `pick_lifecycle_events` table.

---

## 2. `audit_log.entity_id` vs `audit_log.entity_ref`

`audit_log` is the cross-cutting audit table for promotion, outbox, and settlement events. Its two entity columns are asymmetric and easy to get wrong.

| Column | Meaning |
|---|---|
| `audit_log.entity_id` | **FK to the primary entity of the logged event** — e.g. `pick_promotion_history.id`, `distribution_outbox.id`, `settlement_records.id`. It is **not** the pick id. |
| `audit_log.entity_ref` | **The pick id as text.** A denormalized reference to the pick involved, regardless of which primary entity triggered the row. |

**Consequence:** to find all audit rows for a given pick, filter by `entity_ref = <pick_id>`, not by `entity_id`. To join to the primary entity (e.g. the promotion history row), join on `entity_id`.

This asymmetry is deliberate — it lets a single audit table cover events across multiple primary entities while still being pick-searchable — but it is trip-worthy every time.

## 3. `submission_events.event_name` — not `event_type`

The event discriminator column on `submission_events` is **`event_name`**, not `event_type`.

- Column: `submission_events.event_name`
- Values: domain-scoped event identifiers emitted by the submission pipeline

There is no `event_type` column on this table. References to `submission_events.event_type` are bugs.

---

## 4. `settlement_records.corrects_id` — corrections never mutate originals

Settlement corrections are additive, not destructive.

- Column: `settlement_records.corrects_id`
- Semantics: self-referencing FK to the `settlement_records` row being corrected
- Rule: **the original row is never mutated.** A correction inserts a new row with `corrects_id` pointing at the original; the original stays as-is for audit.

**Consequence:** when computing current settlement state for a pick, walk forward to the newest row in the `corrects_id` chain. Do not `UPDATE` existing settlement rows. Do not treat the original as the current state when a correction exists.

---

## 5. `picks.source` — typed union, includes `'system-pick-scanner'`

`picks.source` is a typed union, not free text. The set of permitted values is owned by `pickSources` in `packages/contracts/src/submission.ts`.

- Permitted values include `'system-pick-scanner'` (added in Phase 1, commit `66c9cc1`) alongside the existing `'smart-form'`, operator, and API-derived sources.
- New source values must be added to the union in `contracts` first, then runtime writers may use them. Writing an unrecognized source value will fail contract validation at the submission boundary.

---

## 6. `provider_offers.is_opening` — required for CLV and scanner

`provider_offers.is_opening` is the line-tag flag used by CLV calculation and the system pick scanner to identify opening-line rows.

- Column: `provider_offers.is_opening` (boolean)
- Set by: `apps/ingestor` as part of SGO ingest
- Consumer: `system-pick-scanner` reads rows where `is_opening = true`; CLV logic requires it to resolve opening price
- Fix history: populated correctly since UTV2-400; older rows may be backfilled but should not be assumed NULL-safe

If this column is missing or NULL on rows that should have it, the scanner and CLV will silently miss the opportunity. It is a *functional* requirement, not a display flag.

## 7. Tests fall back to in-memory repositories when credentials are absent

This is a schema-adjacent fact that causes wrong assumptions during debugging.

- All API/worker servers instantiate Supabase clients only when `SUPABASE_SERVICE_ROLE_KEY` is present.
- When absent, they fall back to `InMemory*` repository implementations from `packages/db/src/runtime-repositories.ts`.
- Unit tests (`pnpm test`) exercise the in-memory path. They do **not** exercise real schema constraints, RLS, triggers, or indexes.
- Runtime proof for T1 requires `pnpm test:db` against live Supabase (see `TRUTH_CHECK_SPEC.md` §4.5).

**Consequence:** a green `pnpm test` run says nothing about whether a migration is applied or whether the schema matches the types. Always run `pnpm test:db` or query Supabase directly when schema correctness matters.

## What belongs here

Add a fact to this doc if **all** are true:

- It is a schema naming or semantic fact (not a data snapshot).
- It is non-obvious — agents or engineers will guess wrong without it.
- It is stable — it has not changed in the last phase and is not expected to change.
- It is frequently needed — it is referenced across multiple lanes or surfaces.
- It cannot be reliably derived by reading `packages/db/src/database.types.ts` in under a minute.

Do not add:

- Migration head counts, row counts, or other volatile snapshots — look these up dynamically.
- Facts that are already self-evident from `database.types.ts` or the contracts package.
- Transient Phase-specific invariants — those belong in the relevant phase contract (e.g. `PHASE2_SCHEMA_CONTRACT.md`).
- Provider knowledge (SGO key formats, grading rules) — those belong in `docs/05_operations/PROVIDER_KNOWLEDGE_BASE.md`.

When a fact here stops being trip-worthy (e.g. because a column is renamed), remove it. This file is an aid, not an archive.
