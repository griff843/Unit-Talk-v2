## 1. The execution chain

```text
CONSTITUTION  (docs/00_constitution/UNIT_TALK_CONSTITUTION_V1.md — supreme)
   │   defines: 14 principles · 19 capability layers · 6 maturity stages · end state
   ▼
PROGRAMS  (§18.3 — P1..P5, one per convergence theme)
   │   each Program = a set of capability layers converging to a maturity stage
   ▼
WORKSTREAMS  (WS-x.y — a slice of a Program, e.g. WS-2.1 Certification Runtime)
   │   each WS owns a coherent runtime surface
   ▼
ISSUES  (UTV2-#### — one issue → one lane → one branch → one PR)
   │   each issue implements or hardens one constitutional requirement
   ▼
PROOF BUNDLES  (docs/06_status/proof/UTV2-####/ — SHA-bound evidence)
   │   mechanical proof the issue satisfies its constitutional requirement
   ▼
CERTIFICATIONS  (certification_records + Linear cert docs — runtime trust state)
       a Program is certified only when its layers' proofs are complete + PM-approved
```

**Rule:** every level derives authority from the level above. An issue with no constitutional requirement is out of scope; a certification with no proof bundle is invalid (§2.11, §10).

## 4. Issue → constitutional requirement binding

Every issue must declare, in its lane manifest, the constitutional anchor it satisfies:
- the **capability layer** (e.g. `4.11 Settlement`),
- the **invariant(s)** it enforces (from that layer's `Required Invariants`),
- the **proof** it will produce (from that layer's `Required Proof`),
- the **certification condition** it advances (from that layer's `Certification Conditions`).

This binding is what makes board generation deterministic: future issues are generated by walking unfulfilled `Required Proof` items per layer.

## 5. Proof → certification gating

Per §10 and §20.6:
1. An issue produces a **proof bundle** (SHA-bound to merge SHA).
2. A workstream is complete when all its issues' proofs validate.
3. A **Program is certified** only when: its layers' `Certification Conditions` are met by proof, PM authority approves (§2.9), and (for runtime layers) burn-in / live proof passes.
4. Certification is **runtime state, not a label** (§4.15) — revocable on stale proof, replay divergence, or invariant violation (§10.6).

## 6. Activation order (constitutional sequencing — §18.2)

```text
truth before capability → eliminate false confidence → runtime enforcement over process
→ immutable truth before replay → replay before certification → governance before scale
→ runtime integrity before AI sophistication → adversarial validation mandatory
→ maturity claims must be earned
```

Therefore the **only legal activation order** is P1 → P2 → P3 → P4 → P5, and within P5: burn-in (P5-B) and adversarial (P5-A) before treasury (P5-C) before capital scaling (P5-D). **No capital activation without certification + burn-in (§4.17).**

## 7. Board generation contract (for future automation)
A board generator MUST:
- read capability layers from the constitution as the requirement source,
- generate at most one issue per unfulfilled `Required Proof` item,
- bind each issue to its layer + invariant + proof + certification condition,
- never generate a capital/treasury (P5-C/P5-D) issue as active while those domains are frozen,
- never mark a Program certified without SHA-bound proof for every layer it owns,
- treat Linear as operational tracking only — the constitution + repo proof are authoritative.
