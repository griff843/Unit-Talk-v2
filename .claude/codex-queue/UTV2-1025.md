## Task

* Linear issue: **UTV2-1025 — Block dispatch-auto preflight bypass in lane tooling — require real token for new lanes**

### Why it matters
## Problem

146 of 212 closed lane manifests have `preflight_token: "dispatch-auto"` — a string literal, not a path to an actual preflight artifact. The lane tooling accepts this string without validation.

While these are all closed lanes (historical), the tooling still allows new lanes to be opened with `dispatch-auto` as the token, bypassing the preflight validation that `ops:lane-start` is supposed to enforce. The `ops/shared.ts:621` validation only checks that `preflight_token` is truthy, not that it points to a real file.

## Fix

In `ops/shared.ts` or `lane-start.ts`:

* Reject `"dispatch-auto"` as a valid preflight token value for new lanes
* Validate that `preflight_token` is a relative path that resolves to an existing file at lane-start time
* If backwards-compat is needed for reading old closed manifests, add the check only on the write path
