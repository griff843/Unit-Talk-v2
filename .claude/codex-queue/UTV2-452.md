## Task packet

**Linear issue:** UTV2-452 â€” Add grading retry queue â€” re-attempt picks with game_result_not_found after 15 min backoff

**Why it matters:**
If `grading-service.ts` runs before the ingestor has written a `game_results` row for a market, the pick is skipped permanently with `game_result_not_found`. No retry. This means picks for games that finish late or where SGO scores come in slowly will never be graded without manual intervention. A retry queue with 15-minute backoff (up to 3 attempts) fixes this.

**Design constraints:**
- No new DB tables (no migrations allowed in this scope)
- Retry state must be in-memory (lives in the cron process)
- Retry state is a `Map<pickId, { attempts: number; retryAfter: number }>` where `retryAfter` is `Date.now() + 15 * 60 * 1000`
- Max 3 retry attempts â€” after 3 failed attempts, reason becomes `grade_skipped_final` (permanent skip, not added back to retry queue)

**Acceptance criteria:**
1. Add `GradingRetryState` type: `Map<string, { attempts: number; retryAfter: number }>`
2. Add `retryState?: GradingRetryState` to `RunGradingPassOptions`
3. In `runGradingPass`, when `game_result_not_found`:
   - Check if pick is in `retryState` and `Date.now() < retryAfter` â†’ skip with reason `game_result_retry_pending`
   - If pick has `attempts >= 3` â†’ skip with reason `grade_skipped_final`, do NOT add back to retry
   - Otherwise: add/update pick in `retryState` with `attempts + 1`, `retryAfter = now + 15min`, skip with reason `game_result_not_found` (first attempt) or `game_result_retry_scheduled` (subsequent)
4. In `grading-cron.ts`:
   - Create a single `retryState: GradingRetryState = new Map()` at the cron runner level (shared across cycles)
   - Pass it to `runGradingPass` via `options.retryState`
   - `runGradingCronCycles` and `startGradingCronLoop` both use this shared state
5. Retry is idempotent: calling `runGradingPass` multiple times with the same `retryState` before `retryAfter` elapses does not increment `attempts`

**Important:** `GradingRetryState` export lives in `grading-service.ts`. The cron creates the Map and owns its lifecycle.
