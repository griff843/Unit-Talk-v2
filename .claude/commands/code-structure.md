# /code-structure

Enforce architectural boundaries when writing, moving, or reviewing code. Apply whenever a change touches package/app boundaries, imports, generated files, or the service/repository split.

**Canonical reference:** `docs/CODEBASE_GUIDE.md`

---

## When this skill applies

Apply automatically when:
- creating a new file (deciding which package/app it belongs in)
- moving code between packages or apps
- adding a new import (especially cross-package)
- reviewing a PR that touches `packages/*` or multiple `apps/*`
- reviewing a Codex return that touches shared packages
- writing anything in `@unit-talk/domain` or `@unit-talk/contracts`
- editing generated types or migration outputs
- considering "I'll just add a helper in this other package"

---

## Package dependency graph (authoritative)

```
@unit-talk/contracts   ← pure types + domain contracts (no runtime deps)
          ↑
@unit-talk/domain      ← pure business logic (imports contracts only)
          ↑
@unit-talk/db          ← DB types, repository interfaces + impls (imports contracts, domain)
          ↑
@unit-talk/config      ← env loading only
          ↑
@unit-talk/observability, events, intelligence  ← supporting packages
          ↑
apps/* (api, worker, operator-web, command-center, smart-form, discord-bot, alert-agent, ingestor)
```

**Dependency laws:**
- Arrows go one direction only. Lower layers never import upward.
- `@unit-talk/contracts` has **zero runtime dependencies** — types only.
- `@unit-talk/domain` imports **only** from `@unit-talk/contracts` and Node.js built-ins (non-I/O only).
- Apps import from packages. Packages **never** import from apps.
- **Apps never import from each other.** `apps/api` does not import from `apps/worker`, etc.

---

## Purity rules for `@unit-talk/domain`

`@unit-talk/domain` is a pure business-logic package. It contains scoring, promotion, lifecycle, and CLV logic.

**Forbidden in domain:**
- any import from `@unit-talk/db`
- any import from `@unit-talk/config`
- any import from `@unit-talk/observability`
- any import from `apps/*`
- any HTTP library (`fetch`, `axios`, `got`, `undici`)
- any DB client (`@supabase/*`, `pg`, `postgres`, any ORM)
- `async` functions that do I/O
- `process.env` access
- file system access
- network access
- mutation of external state

**Allowed in domain:**
- pure functions taking data in, returning data or decisions out
- Node.js built-ins that do not perform I/O
- imports from `@unit-talk/contracts`
- pure async (e.g., `Promise.all` on pure computation) — but not I/O

If a domain function needs data, the **app passes the data in**. Domain never fetches.

---

## App ownership rules

**Apps own side effects.** All I/O — DB writes, HTTP calls, Discord interactions, file I/O, cron scheduling — happens in `apps/*`, never in `packages/*`.

**Apps never import from each other.** If two apps need shared logic:
1. The shared logic belongs in a package.
2. If it is pure, put it in `@unit-talk/domain`.
3. If it is a repository/DB concern, put it in `@unit-talk/db`.
4. If it is a type or contract, put it in `@unit-talk/contracts`.
5. If none of the above fit, it is probably not shared — re-examine.

**`apps/api` is the only canonical writer to the database.** Other apps read through `apps/api` or through repository interfaces. The worker polls and records, but all business-critical writes go through the API service layer.

---

## Repository / service boundary

Services are pure functions that receive **repository bundles** as arguments. They do not instantiate DB clients directly.

**Rules:**
- Services take `{ repos: RepositoryBundle }` (or equivalent) as input.
- Services never call `supabase.from(...)` directly.
- Repository interfaces live in `packages/db/src/repositories.ts`.
- Implementations (`InMemory*`, `Database*`) live in `packages/db/src/runtime-repositories.ts`.
- Handlers coerce raw requests, pass to controllers, controllers call services with repos.
- All servers **fall back to in-memory repositories** when Supabase credentials are absent — this is how unit tests run without a live DB. Do not break this pattern.

**Forbidden:**
- a service importing `@supabase/supabase-js` directly
- a handler calling a DB client directly
- a repository implementation leaking DB-specific types across the interface
- business logic in repository implementations (repositories move data; services make decisions)

## Generated-file rules

- `packages/db/src/database.types.ts` is generated by `pnpm supabase:types`. **Never hand-edit.** If it drifts, regenerate.
- Migration SQL files under `supabase/migrations/` are authored manually, but numbered serially. Do not reuse numbers.
- `.js`, `.d.ts`, and `.map` files under any `src/` directory are build artifacts. They must not be checked in. If they appear, they are a build misconfiguration — fix the config, do not commit the output.
- TypeScript is built via project references. Do not add standalone `tsc` invocations that bypass the project graph.

---

## Contract-first rule

New cross-package types originate in `@unit-talk/contracts` **before** implementation.

- If a type is used in more than one package → it belongs in `contracts`.
- If a type is consumed by apps and domain → it belongs in `contracts`.
- If a type is purely internal to a single package → it stays local.
- Promotion policies, scoring profiles, target registry, and member tier definitions all live in `contracts`. Do not duplicate them elsewhere.

When adding a new type, decide its home first. Moving it later is more expensive than placing it correctly.

---

## File-placement decision tree

```
Is it a type used by more than one package?
  → @unit-talk/contracts

Is it a pure business decision (scoring, lifecycle, promotion, CLV)?
  → @unit-talk/domain

Is it a DB repository interface or implementation?
  → @unit-talk/db

Is it a runtime side effect (DB write, HTTP, Discord, file I/O)?
  → apps/<the relevant app>

Is it a handler / controller / route?
  → apps/api (or the owning app)

Is it a worker poll / delivery adapter?
  → apps/worker

Is it shared by two apps?
  → STOP. It belongs in a package, not in either app.
```

---

## Anti-patterns to reject

Reject and flag any of these on sight:

- `packages/domain/**/*.ts` importing `@supabase/*`, `pg`, `@unit-talk/db`, or `@unit-talk/config`
- `packages/domain/**/*.ts` using `process.env`
- `packages/domain/**/*.ts` declaring `async` functions that touch network or disk
- `apps/<a>/src/**` importing from `apps/<b>/src/**`
- Hand-edited `database.types.ts`
- New package created without a clear dependency story (no justification in the PR description)
- A scoring weight defined in `apps/api/src/**`
- A service calling `createClient(...)` directly
- A repository implementation containing business rules (thresholds, decisions, filtering by policy)
- A new "utility" file that duplicates logic already in `@unit-talk/domain`
- `.js` / `.d.ts` / `.map` files committed under `src/`
- A new top-level directory introduced without explicit PM approval
- A route or cron job added to an app that is not its canonical owner

## Phase-specific rules (Phase 2)

These are hard invariants until Phase 3 opens. Enforced in addition to the rules above.

- No writes to `picks` from the candidate layer — `pick_candidates.pick_id` must remain NULL.
- `pick_candidates.model_score / model_tier / model_confidence` must remain NULL in Phase 2 code.
- `pick_candidates.shadow_mode` must default `true` and **must not be set `false`** in Phase 2.
- The materializer writes `market_universe` rows only.
- The board scan writes `pick_candidates` rows only.
- `system-pick-scanner` is a parallel path — it does **not** route through `market_universe` or `pick_candidates`.
- `UTV2-459` and `UTV2-460` migrations must **not** merge in the same deploy — serial merge only.

Full Phase 2 contract: `docs/02_architecture/PHASE2_SCHEMA_CONTRACT.md`.

## Verification grep patterns

These patterns can be run manually or wired into CI. Each must return zero matches.

```bash
# Domain must be pure
grep -rE "from '@unit-talk/(db|config|observability)'" packages/domain/src/
grep -rE "from '(pg|@supabase/)" packages/domain/src/
grep -rE "process\.env" packages/domain/src/

# No cross-app imports
grep -rE "from '(\.\./)+apps/" apps/
grep -rE "from 'apps/" packages/

# No hand-edits to generated types (flag if modified outside supabase:types runs)
git log -1 --format=%s packages/db/src/database.types.ts

# Phase 2 invariants
grep -rE "pick_candidates.*pick_id.*=.*[^NULL]" apps/ packages/
grep -rE "shadow_mode\s*=\s*false" apps/ packages/
```

## Red flags — stop if you see these

- A new file placed "because the other files are here" without checking the decision tree
- A service instantiating a Supabase client
- A scoring weight or threshold living outside `@unit-talk/domain`
- An app calling another app's internals
- `@unit-talk/domain` acquiring any runtime dependency
- `database.types.ts` showing in a diff that is not a `supabase:types` regeneration
- A new package proposed with overlapping responsibility with an existing one
- A fallback path that silently switches from the in-memory repository to a real client in a test context
- Phase 2 boundary violation in any diff

Report the violation and stop. Do not implement "around" the boundary.
